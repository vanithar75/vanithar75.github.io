The impact of disaggregation from edge 2 cloud provides an abundance of choices for compute storage and networking for user applications. Compute is now a commodity infrastructure. With the rapid evolution of automation tools that abstract the deployment complexity of infrastructure and network configuration, the future promises an extension of the cloud provider model, where compute resources are available as a fabric that can be utilized as and when required.

The edge platforms are vendor agnostic and application-centric and provide a compelling alternative option to process data closer to the source for workloads that are compute (latency sensitive) & network intensive (i.e., to reduce the load on the network) in a secure private network.

In this blog post, I will provide some perspective on how edge platforms can help to transform traditional connectivity infrastructure into a global digital infrastructure service.

Designing Products and Services for global deployment

Connectivity is an integral part of any next-generation product design and most products and services are designed to go global. Hence there is a need to provision for global deployment and operations while designing new products. Edge platforms provide opportunities to develop smart systems that are optimized for local operations and global efficiency.

From an enterprise customer perspective from Edge 2 cloud, we need the right tools to deploy and manage the next-generation applications that are distributed in nature. Having a unified control plane to orchestrate services reduces the global deployment management complexity.

Edge platforms provide choice, and flexibility to de-couple and simplify the distributed edge services development providing tools to manage the infrastructure, networking, and security for applications.

Low-latency compute services

Though there are plenty of options for low-latency computing services the trade-off is price vs performance.

The enterprise edge platform offers the best performance however the cost of the solution is much higher, as this involves deploying a network function i.e., UPF - User Plane Function at the edge or enterprise customer premise.

The near-edge and cloud-edge platforms offer the next best performance and an alternative option to the public cloud.

For distributed edge services development we now have multiple new locations and infrastructure options available (far edge, near edge, cloud edge) besides the public cloud.

From a telecom networking terminology perspective, the edge 2 cloud continuum is classified as last-mile, middle-mile, and long-haul connectivity. A wide range of network technologies is used from Edge 2 cloud as indicated in the link [ 1 ]. The edge platform on the middle mile transport infrastructure is a.k.a cloud edge by the LFEdge community (e.g., Interconnect providers viz. Equinix).

Like the network edge, and enterprise edge platforms, the cloud edge platform also provides computing, and storage for user applications at Interconnect locations. This is provided by Independent Software Vendors (ISV) jointly with their Interconnect provider partners viz., Equinix as in the below link.

These solutions enable seamless connectivity from Edge 2 cloud and can help to accelerate the global deployment of edge-enabled solutions.

Multi-Vendor Edge Platforms - Similarities and Differences

The edge 2 cloud continuum has multiple management domains, and the edge platforms try to simplify the management complexity across multiple domains and provide automation and observability tools for end 2 end visibility.

Among the three multi-vendor edge platforms that I wrote about in my earlier post [ 1 ], each of the edge platforms takes a slightly different approach to edge orchestration. There are some similarities and differences between them.

· Reduce the complexity of provisioning, security, and networking. Simplify application and network function deployment, and policy enforcement. i.e., unified policy-driven routing and networking across multiple domains (multiple types of edges. i.e., near edge, far edge, cloud edge).

· Easy integration to Kubernetes clusters from on-premises or private or public cloud.

· In terms of usability all the edge platforms have a GUI-based editor (JSON/YAML-based declarative specification) and APIs to automate with the NetDevOps pipeline enabling intelligent workflows.

* As I mentioned in my earlier post [ 1 ] edge enabled solution development requires an independent set of stakeholders to offer industry solutions that provide competitive differentiation. There is no technical requirement to tie in the industry solutions to the edge platforms. Edge platforms are means to simplify the distributed data processing needs.

The differences don’t really mean the strengths or weaknesses of the edge platform as these platform capabilities are built around the core competency of the platform vendors.

Impact of the edge platform differences on the Customers

These are design choices by the enterprise or the solution providers. In some scenarios pre-integrating, the global connectivity would fit if the enterprise wants a fully managed service offering. If an enterprise wants to have control of the transport path or optimize the transport cost, edge platforms without pre-integrated connectivity would be a better fit.

This provides a great opportunity for the Communication Service Providers(CSP), and Mobile Network Operators (MNO) to pre-integrate the existing connectivity products with compute services and provide a choice of digital infrastructure to enterprise customers. This shows that the software-defined infrastructure evolution has less dependency on the Hyperscaler solution offering (Full stack hyper scaler propositions for the edge) to handle the management complexity from edge 2 cloud. We now have a new set of emerging solutions as a substitute for Hyperscaler offerings .

Impact of Software-Defined Networking on Edge Service Delivery

As more and more services are now software-defined the management of these disparate domains is now shifted from a vertical to a horizontal manner. i.e., There is no strict necessity to operate each domain independently. Rather the management could be (should be) done horizontally e.g., infrastructure (compute, storage), networking, and security. I would think there would be multiple factors influencing the decision on the number of management domains and the operational model, as this could vary across enterprises depending on the edge products and services and skill set of the teams.

Edge orchestration tools help to deploy applications where they’re needed across heterogenous environments from the edge 2 cloud, taking advantage of the design-by-contract (domain data) model with a microservice-based architecture. This architecture approach also offers the portability of solutions across multiple environments.

Distributed global edge service development

We now have an abundance of compute for the application developers from the edge 2 cloud and the edge platforms lowers the barrier to entry for global deployment of edge applications providing more options for global connectivity and compute. Over time these edge platforms shall have additional capabilities viz., Edge site selection and user-defined routing policy, to manage the price across the various compute environments making compute a utility that an application developer would consume.

On the other hand, with the telecom network capabilities being exposed thru APIs ( Open network APIs) it is possible to create network-driven applications . Federation APIs are also being defined by GSMA for edge resource sharing across mobile network operators. This makes Edge really a fascinating space with plenty of new options on how data can be processed and optimized for distributed data processing and simplify global deployments.

Though these edge platforms are general purpose in nature today over time they will evolve to integrate with the telco network services i.e., converged network services that provide multi-access to enable smart applications .

The edge platforms are a critical enabler to transform the traditional telco connectivity infrastructure to a software-defined platform and MNOs are in the best position to offer the edge 2 cloud digital infrastructure services to accelerate the global edge service deployment.

I hope you gained some perspective on how you could leverage the edge platforms to get started with distributed edge service development.
